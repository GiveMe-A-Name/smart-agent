# 设计补丁：工作流优化与迭代机制

## 概述

本文档是对原始设计文档的补充，针对实际使用中发现的问题提供解决方案。

## 核心设计原则

**灵活引导型流程**：
- 不硬性规定执行顺序，由 Coordinator 根据实际情况判断
- 在关键节点引导用户确认，而非强制流程
- 保持 AI 的创造力和灵活性

---

## 问题 1：Coordinator 跳过 Sub-agents

### 解决方案

Coordinator 在判断需求简单、想跳过 sub-agents 时，**必须先跟用户确认**：

```markdown
## Phase 2: 方案评估

基于您的需求，我认为这个任务相对简单，可以直接进入详细设计阶段。

### 方案概述
[Coordinator 给出的简要方案]

### 确认
- **选项 A**: 直接进入详细设计阶段（推荐）
- **选项 B**: 启动完整的研究评估流程

请确认您希望如何继续？
```

### Coordinator 的判断因素

Coordinator 可以根据以下因素**自主判断**是否建议跳过：
- 需求复杂度
- 预估工作量
- 技术栈熟悉度
- 用户明确要求

**注意**：判断后必须询问用户，用户同意后才能跳过。

## 问题 2：Researcher 和 Evaluator 协作流程

### 解决方案：明确的交付机制

```
Researcher 完成研究 → 交付给 Coordinator → Coordinator 转发给 Evaluator → Evaluator 评估
```

### Coordinator 的职责

1. **收集 Researcher 产出**：确保 Researcher 输出已就绪
2. **格式检查**：验证是否符合 Researcher 输出格式要求
3. **转发给 Evaluator**：附带用户原始需求作为评估基准
4. **处理评估结果**：
   - 通过 → 进入下一阶段
   - 拒绝 → 将反馈转回 Researcher

---

## 问题 3：Researcher 输出格式标准化

### 解决方案：明确 Researcher 职责边界

Researcher 的输出应该是**简洁的可行性方案**，而非详细设计：

```markdown
## 研究结果

### 方案概述
[1-2 句话概括推荐方案]

### 技术路线（可选 1-2 个）
- **方案 A**: [技术路线描述]
- **方案 B**: [备选路线]

### 关键风险点
- [风险点 1]
- [风险点 2]

### 推荐决策方向
[为什么推荐这个方向]
```

### 关键原则

- **不包含**：具体代码实现、文件结构、详细 API 设计
- **只包含**：方案可行性、风险评估、决策建议
- 详细设计是 Planner 的职责

## 问题 4：Evaluator 反馈迭代闭环

### 解决方案：默认启用 2-3 轮迭代

```
Researcher 输出 → Evaluator 评估
                    ↓
              [通过] → 进入下一阶段
                    ↓
              [拒绝] → 反馈给 Coordinator
                    ↓
              Coordinator 转给 Researcher
                    ↓
              Researcher 修订 → 再次评估 (循环)
```

### 迭代控制

- **最大迭代次数**：2-3 轮（默认值，可配置）
- **达到上限后的处理**：
  - 报告当前状态和分歧点
  - 由用户决定：继续、调整方向、或放弃

### Evaluator 反馈格式

```markdown
## 评估结果：需要修改

### 问题
1. **[问题类别]**: [描述]
   - 影响：[对方案的影响]
   - 所需操作：[需要 Researcher 做什么]

### 修改方向
- [具体、可操作的修改建议]
- [避免模糊的反馈]
```

## 补充设计内容

### 1. 阶段准入/准出规则

| 阶段 | 准入条件 | 准出条件 |
|------|---------|---------|
| 头脑风暴 | 用户发起 | 用户确认需求 |
| 研究评估 | 用户确认需求 | Evaluator 通过 或 达到迭代上限 |
| 计划审核 | Evaluator 通过 | Reviewer 通过 或 达到迭代上限 |
| 用户确认 | Reviewer 通过 | 用户确认计划 |
| 编码实现 | 用户确认计划 | 代码完成 |

### 2. 跳过机制

| 场景 | 处理方式 |
|------|---------|
| Coordinator 建议跳过 | 必须用户确认 |
| 用户主动要求简化 | 记录在上下文，继续执行 |
| 迭代超过上限 | 用户决定是否继续 |

### 3. 迭代状态跟踪

Coordinator 维护迭代计数器：
- `research_iterations`: 当前研究迭代次数
- `review_iterations`: 当前审核迭代次数

## 问题 6：Evaluator 评审规则过于严格

### 问题分析

当前设计：Evaluator 只输出"接受"或"拒绝"两种结论，任何问题都导致不通过。

问题：
- 没有根据风险级别区分对待
- 低风险问题（如代码风格）也会阻塞流程
- 增加不必要的迭代成本

### 解决方案：引入风险分级评审

#### 风险级别定义

| 级别 | 描述 | 示例 | 处理方式 |
|------|------|------|----------|
| **高风险 🔴** | 可能导致系统崩溃、数据丢失、安全问题 | 数据损坏、认证绕过、内存泄漏 | 必须修复，阻塞通过 |
| **中风险 🟡** | 功能不完整、性能问题 | 边界情况缺失、可扩展性问题 | Agent 自行判断是否可接受 |
| **低风险 🟢** | 代码风格、文档问题、非关键优化 | 命名规范、缺少注释 | 记录为建议，不阻塞通过 |

#### 通过规则

- **有高风险** → **必须拒绝**（强制）
- **只有中/低风险** → **Agent 自行判断**是否可接受
- 判断因素：问题是否可后续解决、是否为边界情况、是否有变通方案

#### 灵活性原则

评审不应过于死板。Agent 应根据实际情况做出判断：
- 中风险问题如果是边界情况或可后续优化，可以接受
- 中风险问题如果影响核心功能，应该拒绝并要求修订

#### Evaluator 输出格式

```markdown
## 评估结果

### 风险级别分类

#### 高风险 🔴
- [问题 1]: [描述及影响]

#### 中风险 🟡
- [问题 1]: [描述] - [为何可接受/不可接受]

#### 低风险 🟢
- [问题 1]: [描述]

### 最终结论

- **接受** ✅ - 无高风险问题，中风险可接受（说明原因）
  - 或 **需要修改** ⚠️ - 存在高风险问题，或中风险不可接受

### 建议改进（不阻塞流程）
- [低风险问题列表，供参考]
```

#### 评审示例

**场景：有中风险但整体可接受**

```markdown
## 评估结果

### 风险级别分类

#### 高风险 🔴
（无）

#### 中风险 🟡
- **性能考虑**: 大文件上传可能较慢
  - 可后续通过分片上传优化
  - 当前方案对 MVP 阶段可接受

#### 低风险 🟢
- 文档示例可以更简洁
- 变量命名可以更具描述性

### 最终结论

- **接受** ✅

### 说明
无高风险问题。中风险可在后续迭代优化，当前阶段可接受。

### 建议改进（不阻塞流程）
1. 后续考虑分片上传优化
2. 使用更具语义化的变量名
```

**场景：存在高风险**

```markdown
## 评估结果

### 风险级别分类

#### 高风险 🔴
- **数据一致性问题**: 分布式事务处理缺少回滚机制，可能导致数据损坏

#### 中风险 🟡
- **性能风险**: 大数据量查询未考虑分页

#### 低风险 🟢
- 注释不够详细

### 最终结论

- **需要修改** ⚠️

### 必须修复
1. 添加分布式事务的补偿机制或回滚方案

### 建议改进（不阻塞流程）
- 实现查询分页
- 补充关键逻辑的注释说明
```
