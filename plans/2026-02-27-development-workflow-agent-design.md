# Development Workflow Agent 设计文档

## 概述

本文档描述一个全栈型开发工作流 Agent，能够帮助用户完成从问题讨论到代码实现的完整开发流程。

## 整体架构

```
┌─────────────────────────────────────────────────────────────────────────┐
│                            主 Agent                                      │
│  - 主持头脑风暴                                                           │
│  - 管理上下文分发                                                          │
│  - 用户交互                                                                │
│  - 流程控制                                                                │
└─────────────────────────────────────────────────────────────────────────┘
         │                    │                    │
         ▼                    ▼                    ▼
   ┌──────────┐        ┌──────────┐        ┌──────────┐
   │ Agent A  │        │ Agent C  │        │ Agent E  │
   │ (研究)   │        │ (规划)   │        │ (编码)   │
   └────┬─────┘        └────┬─────┘        └────┬─────┘
        │                   │                   │
        ▼                   ▼                   ▼
   ┌──────────┐        ┌──────────┐        ┌──────────┐
   │ Agent B  │        │ Agent D  │        │ Agent F  │
   │ (评估)   │        │ (审核)   │        │ (Review) │
   └──────────┘        └──────────┘        └──────────┘
```

## 阶段流程

### 阶段 1: 头脑风暴

**参与者:** 用户 ↔ 主 Agent

**流程:**
1. 用户提出问题或需求
2. 主 Agent 使用 brainstorming skill 与用户讨论
3. 明确需求细节
4. 用户确认需求

### 阶段 2: 评估解决方案

**参与者:** Agent A (研究) + Agent B (评估)

**流程:**
1. 主 Agent 将用户确认的需求分发给 Agent A 和 Agent B
2. Agent A 联网搜索研究解决方案
3. Agent B 评估 Agent A 的产出:
   - 自行思考评估标准，发送给主 Agent 让用户确认
   - 接受用户确认的评估标准
   - 给出接受/拒绝结论及改进方向
4. 如果 Agent B 拒绝:
   - Agent B 将改进方向发送给 Agent A
   - Agent A 根据改进方向优化方案
   - 返回步骤 3 重新评估
5. 循环直到 Agent B 接受

### 阶段 3: 审核规划

**参与者:** Agent C (规划) + Agent D (审核)

**前置:** 阶段 2 产出的解决方案

**流程:**
1. 主 Agent 将解决方案分发给 Agent C 和 Agent D
2. Agent C 阅读仓库代码，理解上下文
3. Agent C 根据解决方案制定具体规划（代码/设计）
4. Agent D 审核规划:
   - 接受原始解决方案 + Agent C 的规划
   - 评估规划是否完整覆盖解决方案
   - 给出接受/拒绝结论及改进方向
5. 如果 Agent D 拒绝:
   - Agent D 将改进方向发送给 Agent C
   - Agent C 根据改进方向优化规划
   - 返回步骤 4 重新审核
6. 循环直到 Agent D 接受
7. 主 Agent 将规划输出给用户确认

### 阶段 4: 用户确认规划

**参与者:** 用户 ↔ 主 Agent

**流程:**
1. 主 Agent 向用户展示最终规划
2. 用户确认或提出修改意见
3. 如果用户提出修改:
   - 主 Agent 判断修改性质（小改/大改）
   - 小改: 复用原 Agent C/D 继续迭代
   - 大改: 启动新的 Agent C/D
   - 返回阶段 3 重新审核
4. 用户确认后进入下一阶段

### 阶段 5: 编码实现

**参与者:** Agent E (编码) + Agent C (技术支持)

**前置:** 审核通过的规划

**流程:**
1. 主 Agent 启动 Agent E 进行编码
2. Agent E 可向 Agent C 提问，获取:
   - 代码上下文
   - 规划细节
   - 设计意图
3. Agent E 列出编码 todos
4. Agent E 按计划编码
5. Agent E 运行基本测试验证实现
6. 完成后交付代码给主 Agent

### 阶段 5.5: 性能测试（可选）

**参与者:** Agent G (性能测试) + Agent E/A/C (技术支持)

**前置:** 阶段 5 完成的代码

**判断条件:** 主 Agent 根据头脑风暴内容、Agent B 的解决方案、Agent C 的规划判断是否可量化

**可量化的指标示例:** 响应时间、内存占用、FPS、吞吐量等

**流程:**
1. 主 Agent 判断是否可量化
   - 如果不可量化: 跳过阶段 5.5，直接进入阶段 6
   - 如果可量化: 继续以下流程
2. Agent G 在编码前跑 baseline 测试，记录原始性能数据
3. Agent E 编码完成后
4. Agent G 再次跑性能测试，对比 baseline
5. 评估结果:
   - 性能有改善 → 进入阶段 6
   - 性能未改善或变差 → 继续步骤 6

6. 性能未改善时的处理:
   - 主 Agent 启动 Agent H 探究根因
   - Agent H 可向 Agent E/A/C 提问，获取:
     - 解决方案细节
     - 代码变更内容
     - 性能瓶颈分析
   - Agent H 输出根因分析报告

7. 主 Agent 展示根因给用户，用户选择:
   - **全部重启**: 回到阶段 2，重新评估方案（带着根因作为背景信息）
   - **局部重试**: Agent H 提出修复方向 → Agent E 尝试 → Agent G 再次测试
   - 循环直到性能改善或用户放弃

### 阶段 6: Code Review

**参与者:** Agent F (Review) + Agent E (技术支持)

**前置:** 
- 阶段 5 完成的代码（不可量化时）
- 阶段 5.5 通过的代码（可量化时）

**流程:**
1. 主 Agent 启动 Agent F 进行 Code Review
2. Agent F 可向 Agent E 提问，获取:
   - 代码上下文
   - 实现细节
3. Agent F 进行全面 review
4. 如果发现问题:
   - Agent F 将问题反馈给 Agent E
   - Agent E 修复问题
   - 重新 review
5. 循环直到 Agent F 通过
6. 流程结束

## 角色定义

| Agent | 角色 | 主要职责 |
|-------|------|----------|
| 主 Agent | 协调者 | 头脑风暴、上下文分发、用户交互、流程控制 |
| Agent A | 研究者 | 联网搜索，研究解决方案 |
| Agent B | 方案评估者 | 评估方案完整性、可行性，给出改进方向 |
| Agent C | 规划者 | 阅读代码，制定实施规划 |
| Agent D | 规划审核者 | 审核规划是否覆盖解决方案 |
| Agent E | 编码者 | 实现代码，运行基本测试 |
| Agent F | Code Reviewer | 代码审查，确保质量 |
| Agent G | 性能测试者 | 跑 baseline 和对比测试，评估性能改善 |
| Agent H | 根因分析者 | 分析性能未改善的原因，提出修复方向 |

## 上下文传递机制

### 原则
- 主 Agent 维护 `context_map`，记录每个子 Agent 已获取的信息
- 派发新任务时，只传递增量信息
- 已读取的共同信息（如用户确认的需求）只需传递一次

### 实现方式
- 主 Agent 中转模式
- 主 Agent 预先整理上下文，一次性分发给对应 Agent
- 避免子 Agent 直接通信带来的复杂性

### 根因传递
- 根因分析结果持久化存储
- 全部重启时，根因作为背景信息传递给 Agent A/B
- 局部重试时，根因传递给 Agent E/H

## 迭代退出条件

| 阶段 | 退出条件 |
|------|----------|
| 阶段 2 (评估方案) | Agent B 接受方案 或 达到最大迭代次数 |
| 阶段 3 (审核规划) | Agent D 接受规划 或 达到最大迭代次数 |
| 阶段 4 (用户确认) | 用户确认规划 |
| 阶段 5 (编码) | Agent E 完成编码并通过基本测试 或 达到最大迭代次数 |
| 阶段 5.5 (性能测试) | 不可量化时跳过 / 可量化且性能改善 |
| 阶段 5.5 (根因处理) | 用户选择全部重启或局部重试后达成目标，或用户放弃 |
| 阶段 6 (Review) | Agent F 通过 Review 或 达到最大迭代次数 |

### 最大迭代次数

- 每个审核角色（Agent B/D/F）最多迭代 3 次
- 3 次后仍未通过，主 Agent 汇报当前状态和分歧点
- 由用户决定：
  - 继续迭代（再尝试 1-2 次）
  - 调整方向（修改方案/规划/代码）
  - 放弃该阶段

### 超时机制

- 单个阶段设置超时限制（如 30 分钟）
- 超时后主 Agent 汇报进度：
  - 已完成的工作
  - 当前卡点
  - 预估剩余时间
- 由用户决定：
  - 继续等待
  - 调整方向
  - 跳过该阶段（如果可行）

## 用户交互模式

### 信息聚合
- 在关键节点（阶段转换），主 Agent 汇总各 Agent 输出
- 用户看到一个完整的阶段报告

### 问题过滤
- 子 Agent 的问题先发送给主 Agent
- 主 Agent 使用 brainstorming skill 判断:
  - 能自行回答: 主 Agent 回复
  - 需要用户确认: 向用户提问

### 用户反馈处理
- 用户可以随时打断流程
- 修改意见根据性质决定:
  - 小改: 复用原 Agent 继续
  - 大改: 启动新 Agent

### 性能失败处理
- 主 Agent 展示根因分析结果
- 用户选择:
  - 全部重启: 带着根因回到阶段 2
  - 局部重试: Agent H 给出修复方向，继续尝试

## 复用 vs 新启动决策

| 场景 | 决策 |
|------|------|
| Agent B/D 评估被拒绝后的改进迭代 | 复用 |
| 用户提出小修改 | 复用 |
| 用户提出方向性大改动 | 新启动 |
| 编码阶段开始 | 新启动 Agent E |
| 性能测试阶段开始 | 新启动 Agent G |
| 根因分析阶段开始 | 新启动 Agent H |
| Review 阶段开始 | 新启动 Agent F |

## 可量化判断

### 判断依据
主 Agent 综合以下信息判断是否可量化:
1. 阶段 1 头脑风暴中用户提到的指标
2. Agent B 解决方案中明确的性能目标
3. Agent C 规划中可测量的指标

### 常见可量化指标
- 响应时间 / 延迟
- 内存占用
- CPU 使用率
- 吞吐量 (QPS/TPS)
- FPS / 帧率
- 文件大小
- 编译时间
- 测试覆盖率

## 后续扩展

本设计可根据实际使用效果进行扩展:
- 支持更多类型的测试（安全测试、压力测试）
- 添加自动化指标监控
- 支持自定义评估标准模板
- 增加根因分析的自动化程度

## MVP 版本规划

### 建议实现顺序

1. **MVP 版本**（阶段 1-4）
   - 头脑风暴
   - 方案研究 + 评估
   - 规划制定 + 审核
   - 用户确认规划

2. **完整版本**（阶段 5-6）
   - 编码实现
   - 性能测试（可选）
   - Code Review

### Agent 合并考量

- MVP 阶段可以先不分 Agent B/D，合并为一个审核角色
- 等 MVP 验证效果后，再根据需要拆分为独立角色
- 当前设计保留拆分结构，便于后续扩展

## Skills 加载机制

### 原则
- 不硬编码固定的 skills 列表
- 主 Agent 根据用户实际拥有的 skills 动态决定加载哪些
- 根据任务性质，让 sub-agents 加载适合的 skills 提高成功率

### 主 Agent 启动时

1. **探索用户拥有的 skills**
   - 主 Agent 列出可用的 skills
   - 识别每个 skill 的用途和适用场景

2. **加载初始 skill**
   - 如果用户有适合头脑风暴的 skill，立即加载
   - 如果没有，主 Agent 使用原生能力进行需求探索

### 创建 sub-agents 时

主 Agent 根据任务性质，动态决定每个 sub-agent 的 skills：

**决策原则：**
- **必须加载**：没有这个 skill 任务无法完成
- **建议加载**：有这个 skill 可以提高任务质量
- **可选加载**：根据任务具体情况决定

**示例映射（根据实际可用的 skills 调整）：**

| 任务类型 | 必须加载 | 建议加载 | 说明 |
|----------|----------|----------|------|
| 研究方案 | - | - | 主要是联网搜索能力 |
| 评估方案 | - | systematic-debugging | 分析方案问题时 |
| 制定规划 | writing-plans | python-expert / typescript-expert | 根据代码库语言 |
| 编码实现 | test-driven-development | python-expert / typescript-expert | 根据代码库语言 |
| 性能测试 | - | verification-before-completion | 验证测试结果 |
| 根因分析 | systematic-debugging | - | 调试分析能力 |
| Code Review | code-reviewer | github | 代码审查能力 |

### Skills 可用性检查

在加载 skill 之前，主 Agent 需要确认：
1. 用户是否拥有该 skill
2. 该 skill 是否适用于当前任务
3. 加载该 skill 是否会提高任务成功率

### 动态调整

- 主 Agent 可以根据任务执行过程中的情况，追加加载额外的 skills
- 如果某个 skill 不适用，主 Agent 可以忽略它并使用原生能力
