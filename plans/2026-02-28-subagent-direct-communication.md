# 设计补丁：Sub-agent 直接通信机制

## 概述

本补丁描述了 sub-agent 之间的直接通信机制，替代原有的"主 Agent 中转模式"。

## 核心设计原则

- **轮转对话**：sub-agents 像合作伙伴一样直接沟通
- **主 Agent 观察者**：只监控不介入，只在必要时干预
- **主 Agent 资源池**：提供初始上下文和工具支持
- **可追溯**：对话记录持久化用于回顾和审计

---

## 通信架构

```
┌─────────────────────────────────────────────────────────┐
│                    主 Agent                              │
│  - 观察者：不直接介入对话                                 │
│  - 资源池：提供初始上下文 + 工具支持                      │
│  - 结论确认 + 分歧处理                                    │
│  - 记录对话用于回顾审计                                   │
└─────────────────────────────────────────────────────────┘
                          │
          初始分发（需求/背景）       最终确认
                          │              │
                          ▼              ▼
┌─────────────────────────────────────────────────────────┐
│              Sub-agent 直接通信 (轮转模式)               │
│                                                          │
│   Researcher ←→ Evaluator                               │
│   Planner   ←→ Reviewer                                 │
│                                                          │
│   流通内容：质疑 / 反馈 / 澄清问题                        │
│   限制内容：需求/背景（主 Agent 提供）                   │
│            最终结论（主 Agent 确认）                    │
└─────────────────────────────────────────────────────────┘
```

---

## 通信内容分类

| 内容类型 | 传递方式 | 说明 |
|----------|----------|------|
| 需求/背景信息 | 主 Agent → Sub-agents | 初始分发，之后不再重复 |
| 质疑/反馈 | Sub-agents ↔ Sub-agents | 直接通信的核心内容 |
| 澄清问题 | Sub-agents ↔ Sub-agents | 直接通信的核心内容 |
| 最终结论 | Sub-agents → 主 Agent → 用户 | 需要主 Agent 确认 |

---

## 轮的定义

为避免歧义，明确定义"轮"的计数规则：

```
1轮 = Agent A 提出方案/问题 + Agent B 回应
2轮 = Agent A 修订 + Agent B 再评估
3轮 = Agent A 再次修订 + Agent B 最终评估
```

- **一轮对话**：包含一个 agent 的发言和另一个 agent 的回应
- **迭代上限**：2-3 轮对话（即 4-6 次交互）
- **超时处理**：如果一方超过预定时间（如 5 分钟）无响应，视为该轮结束

## 阶段 2: 研究-评估对话流程

### 启动规则

```
IF 头脑风暴后主 Agent 和用户讨论出初步方案 THEN
    Evaluator 先开始（发起评估）
ELSE
    Researcher 先开始（提出方案）
END
```

### 对话流程

```
主 Agent 初始化
      │
      ▼
┌─────────────────────────────────────────┐
│         轮转对话循环                      │
│                                         │
│  Researcher ────→  Evaluator             │
│     ↑              │                     │
│     │              ▼                     │
│     │         Researcher (修订)          │
│     │              │                     │
│     └──────────────┘                     │
│              (循环 2-3 轮)                │
└─────────────────────────────────────────┘
      │
      ▼
  达成一致 / 达到上限
      │
      ▼
  提交结论给主 Agent
```

### 退出条件

| 条件 | 处理 |
|------|------|
| 达成一致 | 提交给主 Agent 确认，进入下一阶段 |
| 达到迭代上限（2-3轮） | 主 Agent 介入，作为裁判处理分歧 |
| 不可调和分歧 | 主 Agent 介入，评判方案或提交用户裁决 |

## 阶段 5: 编码实现（直接通信模式）

### 通信模式

阶段 5 采用**直接通信模式**，但与阶段 2/3 有所不同：

- **Planner ↔ Coder**：Planner 提供实现指导，Coder 报告进度和遇到的问题
- **Evaluator（可选参与）**：如遇技术决策争议，可邀请 Evaluator 参与讨论

### 触发条件

- 当 Coder 遇到技术选择问题（如：使用库 A 还是库 B）
- 当 Coder 发现需求实现上的歧义
- 当进度严重落后于计划

### 退出条件

| 条件 | 处理 |
|------|------|
| 功能完成 | 提交给主 Agent，进入性能测试阶段 |
| 遇到阻塞性问题 | 主动请求主 Agent 介入 |

---

## 阶段 5.5: 性能测试（观察模式）

### 通信模式

- **Evaluator 主导**：Evaluator 直接与 Coder 沟通测试需求
- **主 Agent 观察**：不主动介入，仅记录对话

### 触发条件

- 性能测试结果未达标
- 需要调整实现方案

---

## 阶段 6: Code Review（直接通信模式）

### 通信模式

- **Reviewer ↔ Coder**：Reviewer 直接提出审查意见，Coder 进行修复
- **Reviewer 主导**：Reviewer 决定是否需要多轮修订

### 退出条件

| 条件 | 处理 |
|------|------|
| Reviewer 批准 | 提交给主 Agent 确认 |
| 达到修订上限（2轮） | 主 Agent 介入做最终决定 |

---

## 主 Agent 观察模式

### 触发条件

主 Agent 的"被动观察"和"主动介入"有明确的触发条件：

#### 被动观察（持续执行）

- 监听 sub-agents 之间的对话流
- 记录对话内容到持久化存储
- 检测对话状态变化（进行中 → 需介入）

#### 主动介入（触发式）

| 触发条件 | 介入方式 |
|----------|----------|
| 达到迭代上限（2-3轮） | 裁判模式：阅读对话记录，评估双方观点 |
| sub-agent 呼叫"安全词" | 立即介入，了解情况 |
| 长时间无响应（>10分钟） | 询问状态，或终止对话 |
| 检测到流程异常 | 评估是否需要干预 |

### 上下文获取机制

主 Agent 不入对话即可获取上下文的方式：

1. **对话日志订阅**：实时接收对话消息（仅用于观察）
2. **按需加载**：介入时加载完整对话记录
3. **状态查询**：可查询当前对话状态（轮数、最后活动时间）

---

## 分歧处理机制

### 主 Agent 介入场景

1. **达到迭代上限**：2-3 轮对话后仍未达成一致
2. **不可调和分歧**：sub-agents 明确表示无法达成共识
3. **流程异常**：一方长时间无响应或偏离主题
4. **安全词呼叫**：sub-agent 主动请求介入

### 主 Agent 介入方式

1. **裁判模式**：阅读对话记录，评估双方观点，给出仲裁建议
2. **升级模式**：将分歧点提交给用户裁决
3. **技术支援**：如需要，可调用技术专家 agent 提供专业意见

### 仲裁原则

- 优先保障用户需求被满足
- 技术判断以 Evaluator/Reviewer 意见为主
- 效率与质量平衡，避免过度迭代

---

## 安全词机制

### 定义

允许 sub-agents 在以下情况下主动呼叫主 Agent 介入：

- 遇到超出权限范围的问题
- 需要用户确认的需求细节
- 与另一 sub-agent 出现不可调和的分歧
- 对话陷入无限循环或偏离主题

### 使用方式

Sub-agent 在消息中包含 `[ESCALATE]` 标签即可触发主 Agent 介入：

```
我注意到这个方案在可扩展性上存在隐患。[ESCALATE]
```

### 主 Agent 响应

- 检测到 `[ESCALATE]` 标签后，立即加载对话记录
- 评估情况后，选择裁判模式或升级模式

---

## 边界场景处理

### 只有一个 sub-agent 活跃

当某些阶段只有一个 sub-agent 时：

- **阶段 5（编码）**：Coder 独立工作，定期向主 Agent 汇报进度
- **报告机制**：每完成一个关键任务，向主 Agent 提交简短状态更新

### 防止对话无限循环

| 防护机制 | 说明 |
|----------|------|
| 迭代上限 | 硬性限制 2-3 轮，达到后必须介入 |
| 循环检测 | 主 Agent 监控对话内容，检测重复模式 |
| 安全词 | 允许 sub-agent 主动退出循环 |
| 状态标记 | 对话状态标记为"需介入"时强制终止 |

## 对话记录

### 持久化存储

- 每轮对话记录存储在 `.opencode/logs/` 目录
- 命名格式：`{phase}_{agents}_{timestamp}_{id}.md`
- 包含：唯一ID、时间戳、参与方、对话内容、结论

### 记录格式

```markdown
# 对话记录：阶段 2 - Researcher/Evaluator

**ID**: conv_20260228_001
**时间**: 2026-02-28 10:30:00
**迭代**: 第 2 轮

## 对话内容

### Researcher
[提出的方案内容]

### Evaluator
[评估意见]

### Researcher (修订)
[修订后的方案]

## 当前状态
- 状态：进行中 / 已达成一致 / 需要裁判介入 / 已呼叫安全词

## 结论
[最终结论，如达成一致]

## 关键修订点
- 第1轮：提出初始方案
- 第2轮：根据评估意见修订了X和Y
```

### 对话记录压缩

为控制存储空间，对话记录采用以下压缩策略：

- **保留关键修订点**：每轮的核心改动
- **去除重复内容**：只记录有变化的上下文
- **摘要模式**：对于长对话，可生成摘要版本

### 使用场景

- 分歧仲裁时，主 Agent 参考对话记录
- 用户回顾时，可查看决策过程
- 流程审计，分析迭代效率

## 与原设计对比

| 方面 | 原设计 | 新设计 |
|------|--------|--------|
| 消息传递 | 主 Agent 中转 | Sub-agents 直接通信 |
| 上下文管理 | context_map 增量传递 | 初始分发 + 对话记录 |
| 迭代效率 | 每次迭代重新传递上下文 | 对话持续，减少 token 消耗 |
| 可追溯性 | 无 | 对话记录持久化（带唯一ID） |
| 分歧处理 | 无明确机制 | 主 Agent 裁判 + 用户裁决 + 安全词 |
| 测试覆盖 | 无 | 单元测试 + 集成测试 + 端到端测试 |
| 阶段覆盖 | 阶段 2-3 | 阶段 2-3, 5, 5.5, 6 全部覆盖 |
| 边界场景 | 无 | 单 agent 活跃、无限循环防护 |
